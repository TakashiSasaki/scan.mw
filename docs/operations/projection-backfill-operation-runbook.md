# Projection Backfill Operation Runbook

This runbook describes the process for preparing and managing a projection backfill operation packet.

**IMPORTANT:** This is NOT backfill execution.

The operation packet generated is local-only. It is a strictly controlled artifact that verifies preconditions, organizes batches, and dictates the checklist for a pilot backfill procedure.

## Preconditions

Before preparing a projection backfill operation packet, ensure:
1. The readiness assessment is strictly `ready-for-backfill-design`.
2. An explicit target list or an existing valid backfill plan is available.
3. You are fully aware that this step does **not** authorize UI read switching or execute the actual backfill on the network.

## Expected Artifact Flow

A successful pilot operation flows through these specific artifacts:

1. Readiness assessment JSON (Input)
2. Explicit target list JSON (Input, alternative to 3)
3. Backfill plan JSON (Input, alternative to 2)
4. Operation packet JSON (Generated by this runbook)
5. Pre-reconciliation response (`batch-XXX-pre-reconciliation.json`)
6. Recompute responses (`batch-XXX-recompute-response.json`)
7. Post-reconciliation response (`batch-XXX-post-reconciliation.json`)
8. Reconciliation report (`batch-XXX-reconciliation-report.json`)
9. Validation bundle (Aggregated proof of safe completion, built via `ops:validate-projection-backfill-operation`)

### Validating Evidence

You can validate the collected artifacts using the validation bundle tool.
`ops:validate-projection-backfill-operation` validates saved local operation evidence against the generated operation packet.

It consumes the operation packet and a saved artifact manifest. It does not call Firebase, does not write data, does not execute backfill, and does not authorize UI read switching.

The validation outcome varies by mode:
* `dry-run-evidence-pass`: Validates dry-run evidence only.
* `manual-write-evidence-pass`: Validates saved manual-write evidence only and still does not authorize UI read switching.

## Safety Gates

* **No broad collection scan**: Backfill targets are fully explicit.
* **No scheduler**: This operation does not involve scheduled cloud execution.
* **No queue**: There is no queue-based backfill recomputation.
* **No UI switching**: This does not modify active read paths or deploy configurations.
* **Manual operator action required**: Using the explicit manual recompute payload generated requires human action to actually trigger Cloud Functions.

## Failure Handling

If any issue arises during generation or the actual operation flow, halt the procedure:

* **Invalid packet**: If packet generation fails, inspect the listed blockers. Do not proceed until they are resolved in the readiness or targets artifacts.
* **Missing artifact**: All expected JSON responses must be saved locally and retained.
* **Post-reconciliation not equal**: If the recomputed post-reconciliation state is not strictly `equal`, the validation bundle will fail.
* **Count mismatch / Target mismatch**: An unexpected discrepancy in target numbers indicates potential race conditions or invalid data. Stop the operation immediately and debug.

## Next Steps

After successfully assembling the validation bundle for the operation packet, further execution requires **future work**, specifically:

* Actual backfill execution design.
* Rollback policy.
* UI read switching gate.

Use `npm run ops:assess-projection-backfill-execution-design` to evaluate saved validation bundles for readiness to begin execution design.

Once assessed, use `npm run ops:prepare-projection-backfill-controlled-execution-design` to assemble the Controlled Projection Backfill Execution Design Packet from the design gate and the operation validation bundle.

Next, use `npm run ops:prepare-projection-backfill-controlled-execution-review-contract` to generate a Controlled Execution Review Contract.

* The design packet and review contract are local-only.
* They do not call Firebase.
* They do not write.
* They do not execute backfill.
* They do not authorize UI read switching.
* `ready-for-execution-design` and `ready-for-controlled-execution-design-review` are not execution approval.
* The review contract solidifies the operator checklist, risk register, and approval boundaries.
